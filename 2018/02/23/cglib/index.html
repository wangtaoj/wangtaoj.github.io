cglib实现动态代理

发表于 2018-02-23 |

cglib简介

cglib是一个功能强大，高性能的代码生成包。动态生成的一个被代理类(委托类)的子类(代理类), 在子类采用方法拦截的技术拦截父类方法的调用。底层依赖于字节码处理框架ASM, 比使用java反射的JDK动态代理要快。

cglib简单使用

主要了解下面几个组件:

  1. Callback接口, 回调接口
  2. MethodInterceptor接口, 继承Callback接口, 拦截父类方法, 此处可以增加自己额外逻辑。
  3. CallbackFilter接口, 起过滤作用, 当有些方法不需要被代理, 可以使用它
  4. Enhancer, 核心类, 用来设置生成代理一系列配置, 并且生成代理类。

通过部分关键源码可以看到代理类会继承被代理类, 并且会为每一个被代理的方法生成两个方法

CGLIB$add$0: 直接调用父类(被代理类)的方法
add: 继承自父类, 子类会重写, 主要逻辑是MethodInterceptor对象是否为空, 不为空调用intercept方法, 否则直接调用父类的方法。

再说说methodProxy的两个方法invokeSuper, invoke方法

要想明白invokeSuper, invoke两个方法, 必须要知道FastClassInfo的4个属性f1, f2, i1, i2到底是什么东西, 可以通过idea调试观察下, 当invokeSuper, invoke方法执行inthis.init()后, 就被初始化了。

f1其实代表的是被代理类DbService, f2代表的是生成的代理类, 那么i1指示的是DbService的add()方法, 而i2指示的是代理类生成的CGLIB$add$0方法。一般说来调用invokeSuper, invoke这两个方法得到的结果是一样的, 通常也会选用invokeSuper方法, 因为不需要传被代理类对象进来, 所以更简洁点。

两者使用区别:

  1. 调用对象不同, 一个是被代理类对象, 另一个是代理对象
  2. 根据第一个不同就可以得到下面这个区别, 即多态。试想在add()方法中又调用了remove()方法。因为调用者分为被代理类对象, 另一个是代理对象, 那么执行invokeSuper在执行add()方法中, 继续调用remove会发生多态现象, 那么就会拦截两次。
    得到的结果就是下面这样:
    invokeSuper:
    开启事务!
    add operator
    开启事务!
    remove operator
    提交事务
    提交事务
    invoke:
    开启事务!
    add operator
    remove operator
    提交事务

总结: invoke方法则与JDK动态代理一样result = methodProxy.invoke(dbService, objects)与
result = method.invoke(dbService, objects)等价的, invoke, invokeSuper虽然这是个很小的点, 但是在事务嵌套中可能出现意想不到的情况。

使用CallbackFilter

有时候被代理类的某个方法不需要被代理, 这时候我们需要过滤这个方法, 此时就轮到CallbackFilter登场了。

开启事务!

提交事务
———–我是分隔线————-
remove operator

说明:

accept()返回值与nhancer.setCallbacks()息息相关, 这里如果被代理的方法是remove方法, 那么与remove方法有关的Callback就是数组中索引为1的callback实例了。其他方法的callback实例为索引为0的对象。NoOp.INSTANCE是一个NoOp匿名对象, NoOp接口继承自Callback。这个对象什么也不做, 起到过滤作用。搜索字节码文件, 可以看到remove方法根本就没有被重写过, 只是继承下来而已。可见如果要重写该方法, 必须要与MethodInterceptor相关联。

总结

JDK动态代理与Cglib动态代理都是实现动态代理的一种方式, JDK动态代理要求被代理类必须实现接口, 而cglib动态代理必须要求被代理类能够被继承, 方法能够被重写, 也就是说不能被final修饰。
